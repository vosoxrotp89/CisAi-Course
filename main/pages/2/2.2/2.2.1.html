Workload Mobility

The ability to dynamically move virtual machines, or servers, between different physical hosts, in or across data centers, has not only changed networks requirements but also the operating paradigm for data centers. Today, most large organizations have multiple data centers globally, and public cloud connections from their data centers. This deployment enables the ability of data center teams to operate the environments as if they are a service provider to their organization. Such teams can dynamically provision resources based on changing operational needs. It becomes irrelevant to the user where the application resides, if performance and business continuity expectations are met.

This flexibility with data center resources has continued with the growth of AI and workload mobility. AI workload mobility involves the transfer of either the AI model, an AI inference application, or data consumed or produced from a model. These methods of movement can be manual, or automated, depending on the services built into the platform. For example, Kubernetes and VMware each have capabilities for manual and automated workload mobility built into their products.

The idea of dynamically moving workloads is commonplace and for good reason. AI workload mobility makes good business sense for the following reasons:

  • Cost Optimization Moving workloads between environments can reduce operating expenses by utilizing cheaper resources (like energy, compute power, and so on) during off-peak hours, or taking advantage of price variations between public cloud providers. Designing for shifting workloads can also prevent public cloud lock-in, AKA, vendor lock-in.

  • Disaster Recovery and Load Balancing In the event of partial or total data center failure, moving workloads ensures business continuity. Operators also have the option of load-balancing demand between data centers, or ensuring the application is hosted in the region closest to the users. This capability is important when the closest proximity to resources changes as time zones come online and offline during their respective business hours.

  • Performance As workloads are moved closer to the sources of users, latency can be reduced, providing better user experiences. Organizations may also use elastic services that automatically scale workloads, both vertically and horizontally, based on demand.

  • Compliance Many organizations are under strict regulations that dictate where data can be stored and accessed. Moving such workloads can help to meet specific security or audit requirements by shifting them to environments with higher security standards or certifications.

  • Optimization Workloads can be moved to environments that are better suited for their specific requirements. In the case of AI workloads, understanding the demands of the training and inference fabrics allows operators to optimize placement. With the advent of edge computing, AI workloads are increasingly deployed at the edge, closer to where data is generated. And since training AI models is resource intensive, AI inference is more suitable for edge locations where the applications may be moved to different edge locations depending on real-time demands.

  • Multi-tenancy Using virtualization techniques that are afforded by SDN solutions, AI resources within fabrics can be carved out and assigned to different business units. The ability to move training jobs or inference applications between sites or within a multi-tenant data center provides flexibility. It allows operators to maximize available resources and to support concurrent AI workloads within a cluster where resources are virtually assigned and instantiated.

AI Workload Mobility Methods

Both manual and automated AI workload mobility are appropriate options depending on the specific needs of an organization. Each method has its advantages, disadvantages, and a set of tools to move workloads.

Manual Methods

Manual approaches to workload mobility provide greater control and customization but at the cost of time and potential errors. Manual methods use data transfer tools such as Secure Copy Protocol (SCP) or even cloud storage services such as AWS S3 or other public cloud storage services. The capability for manually deploying and moving containerized applications is built into platforms like Docker, Rancher, and Kubernetes. Similarly, hypervisor platforms like VMware also support manual migration options that use technologies like vMotion or manual snapshots. It is also possible to manually pull code for deploying models from repositories like GitHub or GitLab to different environments.

Automated Methods

Automated approaches to workload mobility offer efficiency and scalability but require investment in tools and expertise. There are many automation options provided by varying solutions.

Orchestration technologies include platforms like Kubernetes, which automates the deployment, scaling, and management of containerized AI workloads. Tools like Kubeflow extend Kubernetes specifically for the nuances of AI workflows. Red Hats OpenShift, which is built on Kubernetes, is a container orchestration and cluster management tool. And Apache Mesos is a similar, open-source platform for orchestration that can manage AI workloads across distributed systems. Apache Airflow is another, open-source, workflow orchestration tool for programmatically authoring, scheduling, and monitoring data pipelines. Airflow is used to automate ETL (extract, transform, load) processes for data ingestion and preprocessing and is particularly valuable for managing complex workflows.

Apache Spark, on the other hand, is a powerful open-source distributed computing system that provides an in-memory data processing framework for data processing. It is widely depended on for large-scale ETL tasks. It supports a very large set of data transformations and is used for preparing the data that is to be fed into an AI/ML models training process. Spark also integrates well with libraries like MLlib, which provides a comprehensive set of tools for managing algorithms, feature transformation, model evaluation and tuning, and more. It can integrate with various data sources like AWS Simple Storage Service (S3), Hadoop Distributed File System (HDFS), or other big data tools..

Public cloud providers offer suites of tools geared toward the data science and AI/ML communities for ETL services. These tools include AWS Glue, GCPs Dataflow, or Azures Data Factory. There are also vendors such as Databricks and Snowflake, who offer cloud-based data analytics platforms and data warehousing as a service.

VMware Tanzu provides tools for deploying and managing AI workloads across hybrid environments, in which organizations own and operate their own data centers while also using public cloud services. The Tanzu dashboard provides an overview of the resource utilization at the cluster, node, namespace, and container levels. The dashboard provides data that can be used for resource utilization and capacity planning across clusters, nodes, and pods.

Google Anthos provides a similar capability, facilitating mobility across on-premises, GCP (Google Cloud Platform), and other cloud providers. Azure Arc manages AI workloads across data centers, edge locations, and public clouds.

With regard to the edge, NetApp Data Fabric provides a unified data management framework, and the ability to move workloads between on-premises, edge, and cloud. NVIDIA EGX also supports the deployment of AI workloads at the edge and supports mobility by integrating into hybrid cloud networks.

Serverless architectural platforms, also known as Function-as-a-Service (FaaS), such as AWS Lambda, GCP Functions, or Azure Functions, can enable AI inference workloads to be triggered automatically, based on events. FaaS facilitates the mobility of code that executes microservices without manual intervention.

Some AIOps platforms like Splunk or Dynatrace can automate the movement of workloads based on real-time analysis of infrastructure performance and availability. Some AI model management tools like TensorFlow Serving or TorchServe also provide support for automating the deployment of models between on-premises, cloud, and edge. TorchServe is built for PyTorch-based models that can be used to deploy and manage the models in different environments.

Note

It is important to ensure that network services are configured to enable AI workload mobility. Technologies like VXLAN EVPN and SD-WAN enable software-defined instantiation of services on demand, which help to securely isolate and prioritize data flows over large geographies.
