Device Virtualization

Device virtualization abstracts physical hardware devices to create virtual devices. Virtual devices share the same physical hardware, which improves efficiency and reduces costs. Device virtualization also enables better flexibility, scalability, and resource optimization of computing environments including virtual network interfaces, virtual storage devices, and virtualized I/O operations. Software layers such as hypervisors and VM monitors manage all the computing environments. Device virtualization is important for cloud computing, data centers, and modern IT infrastructure, and enables seamless resource allocation and management.

Device Virtualization Components

The main components of device virtualization are the control, data, and management planes. Components also include software partitioning for modular processes and dedicated hardware components for predictable resource allocation.

The following list describes the main components of device virtualization:

  • Control plane: This plane manages how data is routed and organized within a virtualized environment. It allows different virtual devices to have separate control functions, keeping each network isolated and secure.

  • Data (or forwarding) plane: This plane provides forwarding tables and other databases that you can partition to provide data segregation.

  • Management plane: This plane provides well-delineated management environments that can be used independently for each virtual device.

  • Software partitioning: Partitioning provides modular software processes grouped into partitions and dedicated to certain virtual devices, which creates well-defined fault domains.

  • Hardware components: These components provide hardware that is partitioned and dedicated to certain virtual devices, allowing predictable allocation of hardware resources.

Virtualization mechanisms provide separation between data, control, and management planes.

The separation between the control, data, and management planes in device virtualization is crucial to ensure that different functions of a virtualized environment can operate independently and efficiently. This separation helps maintain security, optimize performance, and manage complexity by isolating control decisions, data handling, and management tasks.

One way to achieve this separation is through virtual routing and forwarding (VRF). VRF allows multiple virtual routing instances to operate on a single physical device, which isolates network traffic and routing tables for enhanced security and organization.

Virtual Routing and Forwarding

VRF technology is used in network virtualization to allow multiple routing table instances to exist simultaneously within the same router. A single physical router can support multiple isolated routing domains. Each VRF instance operates independently and has its own routing table. This approach segregates traffic so that packets are only forwarded within the specific VRF instance to which they belong.

To provide logical Layer 3 separation within a Layer 3 switch, the data plane and control plane functions must be segmented into different VRF instances or environments. This process is similar to separating the Layer 2 control and data planes into different VLANs on a Layer 2 switch.

Each VRF instance represents a unique Layer 3 addressing domain, which is associated with a unique Routing Information Base (RIB). Each Layer 3 interface (logical or physical) belongs to only one VRF instance. Each VRF instance contains a separate address space with unicast and multicast route tables for IPv4 and IPv6, which means that routing decisions are independent of any other VRF instance.

VRF virtualization can be represented as a stack of layers with the same functionality but different configurations.

A VRF instance includes the following components:

  • A subset of the Layer 3 interfaces on a Layer 3 switch: Similar to how Layer 2 ports are assigned to a particular VLAN on a Layer 2 switch, the Layer 3 interfaces of the Layer 3 switch are assigned to a VRF instance. Because the fundamental component is a Layer 3 interface, this component includes software interfaces, such as subinterfaces, tunnel interfaces, loopback interfaces, and switch virtual interfaces (SVIs).

  • A routing table or RIB: Because traffic between Layer 3 interfaces in different VRF instances should be separated, a separate routing table is necessary for each VRF instance. This approach ensures that traffic from an interface in one VRF instance cannot be routed to an interface in a different VRF instance.

  • A forwarding table or Forwarding Information Base (FIB): The FIB is downloaded to line cards and used for actual packet forwarding. This table must also be separated by a VRF instance so that there is one FIB for both, IPv4 and IPv6 for each configured VRF instance.

  • Routing protocol instances: To ensure control plane separation between the different Layer 3 VPNs, it is necessary to implement routing protocols on a per-VRF-instance basis. You can run a separate process for the routing protocol in each VRF instance to accomplish this task. You can also use a subprocess or routing protocol instance in a global process that is in charge of the routing information exchange for the VRF instance.

Per-VRF-instance characteristics are as follows:

  • There are independent routing and forwarding decisions.

  • IPv4 and IPv6 unicast and multicast tables are created automatically.

  • The number of programmed routes is limited.
